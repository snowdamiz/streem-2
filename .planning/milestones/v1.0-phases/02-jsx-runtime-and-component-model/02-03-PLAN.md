```typescript
import { effect, createRoot, onCleanup, signal } from '@streem/core'

// createRoot(fn: (dispose: () => void) => T): T
// Creates an owner scope; fn's return value is returned.
// onCleanup() calls inside fn register on this scope.

// effect(fn: () => void | (() => void)): () => void
// Runs fn immediately, re-runs when tracked signals change.
// If fn returns a function, that function is called before each re-run (per-run cleanup).
// NOT the same as onCleanup() — per-run vs. disposal-only.

// onCleanup(fn: () => void): void
// Registers fn to fire on OWNER DISPOSAL (not per effect re-run).
// Use this to register component-level teardown.
```

  <behavior>
    **onMount(fn: () => void | (() => void)): void**
    Cases:
      fn called → executes once after createRoot scope is active
      fn returns cleanup fn → cleanup fires when owner disposes
      component function calls onMount twice → both callbacks fire once each
      signal read inside fn (no reactive deps) → reads snapshot, never re-runs

    **COMP-02 — component-runs-once:**
    Cases:
      signal read directly in component body → snapshot, does not update
      signal read inside JSX accessor (effect) → updates correctly
      "runs once" means component fn invocation count = 1 regardless of signal changes

    **Show component:**
    Cases:
      when={true} (static) → renders children, no fallback visible
      when={false} (static) → renders fallback (or nothing if no fallback)
      when={accessor} where accessor changes true→false → swaps to fallback; child DOM removed
      when changes false→true → swaps back to children; fresh child scope
      child scope disposal: effects inside children fire cleanup when Show switches to fallback
      anchor comment node ("Show") is stable — parent layout not disrupted

    **For component:**
    Cases:
      each=[a,b,c] → renders 3 items in order
      item added (d) → only item d's DOM created; others untouched
      item removed (b) → item b's scope disposed; b's DOM removed; a,c remain
      items reordered [a,c,b] → DOM reordered to match; no items recreated
      duplicate keys → dev warning (or throw); no undefined behavior
      each=[] → renders nothing (or fallback if provided)
      index getter updates on reorder: item that moved from index 0 to index 1
        reports () => 1 from its index getter after reorder
  </behavior>
  <implementation>
    **onMount():**
    Use effect() with zero reactive dependencies. The effect body calls fn() once.
    If fn returns a cleanup function, return that as the effect's cleanup:
    ```typescript
    export function onMount(fn: () => void | (() => void)): void {
      effect(() => {
        return fn() // if fn returns cleanup, effect treats it as per-run cleanup
      })
    }
    ```
    Note: since onMount's effect has no reactive dependencies, it fires exactly once.
    The returned cleanup from fn() fires on owner disposal (via the effect's cleanup mechanism).

    **Show():**
    Pattern from RESEARCH.md (Pattern 5). Key points:
    - Use a comment node as anchor (document.createComment('Show'))
    - Maintain `currentDispose: (() => void) | null` for the current child scope
    - Wrap `when` in a getter if not already a function
    - Inside effect(): call currentDispose?.(), then createRoot for new child
    - Insert new child DOM BEFORE the anchor using anchor.parentNode.insertBefore()
    - onCleanup(() => currentDispose?.()) at Show component level for final teardown
    - Return the anchor comment node from Show()

    DOM management: use a DocumentFragment for clean insertion. Remove previous DOM nodes
    before inserting new ones. Track which nodes belong to current render.

    **For():**
    Pattern from RESEARCH.md (Pattern 6). Key points:
    - Use `Map<string | number, { dispose: () => void; nodes: Node[]; setIndex: (i: number) => void }>` as reconciliation map
    - Wrap `each` in getter if not already a function
    - Inside effect():
      1. Compute new key set
      2. For stale keys: dispose scope, remove DOM nodes, delete from map
      3. For new keys: createRoot → call children(item, () => index), store {dispose, nodes, setIndex}
      4. For existing keys: call setIndex(newIndex) to update the index value
      5. Re-order: insertBefore each item's nodes before anchor in array order
    - Index: use a simple reactive counter via signal() per item — the index signal is updated via setIndex
      Actually: since items can reorder without teardown, use a plain mutable ref per item
      (not a signal, since only the render function sees it via the getter). For Phase 2,
      `() => indexRef.current` is sufficient — no need for signals inside the index mechanism.
    - Return anchor comment node from For()

    **File structure:**
    Create `packages/dom/src/components.ts` with Show, For, onMount.
    The functions should NOT be JSX components themselves — they're called by h() when the
    type is Show/For/onMount (as function components). h() already handles function types
    via createRoot wrapping.

    Wait — there's a nuance: h() wraps ALL function components in createRoot. But Show and For
    manage their OWN child scopes internally. If h() also wraps them in createRoot, there's
    a double-wrapping. Solution: Show and For create their OWN child scopes for their children
    (not the Show/For component itself). The outer createRoot from h() is the Show/For component's
    scope — onCleanup in Show/For fires when that outer scope disposes. This is correct behavior.

    Dev-mode duplicate key detection in For:
    In `if (import.meta.env.DEV)` block: check if key already exists in rows map before inserting.
    If duplicate, console.warn('Streem <For>: duplicate key detected:', k).
