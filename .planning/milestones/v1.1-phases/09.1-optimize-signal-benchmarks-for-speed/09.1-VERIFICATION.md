### Documented Deviation: propagateDirty Snapshot Retained

Plan 01 truth #1 and key link #1 were based on the premise that removing `[...source.subs]`
was safe because `notifying = true` prevents re-entrant calls. The implementation discovered
this was incorrect: `runEffect()` calls `source.subs.delete(effect)` on stale deps during
the effect re-run (synchronously, inside `propagateDirty`), which mutates the Set
mid-iteration. V8's `OrderedHashTable::Shrink` triggered an OOM crash in tests.

The SUMMARY for Plan 01 correctly documents this as a Rule 1 (Bug) auto-fix. The decision
to retain the snapshot is the correct engineering choice. The batchedEffects Set optimization
(truth #2) fully delivers O(1) deduplication in the actual hot path (batch mode).

**Net result:** 2 of the 3 hotspot optimizations planned were delivered. The third
(propagateDirty snapshot removal) was correctly identified as unsafe and reverted.
