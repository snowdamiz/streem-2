<decisions_made>

- Linked list approach chosen over Set: eliminates new Set() per node, [...source.subs]
  spread on every write, and hash overhead. Same approach Preact signals uses internally.
- notifyVersion counter (not a "queued" boolean): handles the subtle case where an
  effect re-subscribes during its own run and its new link appears at the tail of the
  source's subscriber list. Version check skips it on the same propagation cycle.
- Did NOT change the public API (s() vs s.value): function-call overhead is the
  remaining ~1.75x gap vs Preact signal, but changing the API is a breaking change
  requiring a deliberate decision.
- cleanupFns lazy-init: null until first onCleanup() — most effects never call it.
</decisions_made>

<context>
The linked list rewrite was motivated by the user saying streem was "still drastically
slower." The Set-based approach had 3 allocations per signal (node + Set + closure) and
spread overhead on every write. The linked list brings streem signal-with-root to 23M
ops/sec (faster than Solid's 21.9M), computed-with-root to 11.9M (63% of Preact),
effect-with-root to 8.9M (68% of Preact).

The one remaining structural gap is the callable-signal API: s() is a function call,
Preact's s.value is a property getter. Closing it requires API design decision.

BENCHMARKS.md is fully up to date with the linked-list numbers.
</context>
