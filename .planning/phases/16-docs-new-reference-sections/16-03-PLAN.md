<interfaces>
<!-- Key reactive API facts the executor must use accurately in examples. -->

computed() returns a getter function (NOT a Signal):
```typescript
const doubled = computed(() => count.value * 2)
doubled()  // call to read — lazy pull, only recomputes when dependency changed
```

effect() re-runs synchronously on every dependency change and returns a dispose function:
```typescript
const stop = effect(() => {
  console.log(count.value)
  onCleanup(() => console.log('before next run or dispose'))
})
stop() // manual dispose
```

createRoot() scopes reactive ownership — dispose cleans up all children:
```typescript
const root = createRoot(() => {
  const count = signal(0)
  effect(() => console.log(count.value))
  onCleanup(() => console.log('root disposed'))
})
root.dispose() // disposes effect and fires onCleanup
```

onMount() runs after first render and its return value is the cleanup function:
```typescript
onMount(() => {
  const id = setInterval(() => tick.set(Date.now()), 1000)
  return () => clearInterval(id)  // cleanup
})
```

Dev-mode warnings (from packages/core/src/signal.ts):
- computed() outside owner scope → "[Streem] computed() created without an active owner scope. This computation will never be automatically disposed (disposal leak)."
- effect() outside owner scope → same warning for effect
- These warnings fire in DEV only — production builds exclude them

Signal granularity:
- Fine-grained: separate signals per field → only the field's subscribers re-run
- Coarse-grained: one object signal → all subscribers re-run on any field change

1. **computed() vs effect()** — Explain the key tradeoff:
   - Use `computed()` when you need a derived value (lazy pull — only recalculates when read after a dependency changes, zero side effects)
   - Use `effect()` for side effects only (DOM mutations, fetch calls, timers, logging)
   - Anti-pattern: using `effect()` to maintain a derived value in another signal (creates unnecessary intermediate storage and triggers extra re-runs)
   - Show the anti-pattern vs the correct computed approach:
     ```typescript
     // BAD — effect maintaining derived state:
     const doubled = signal(0)
     effect(() => { doubled.set(count.value * 2) })  // runs as side effect

     // GOOD — computed for derived values:
     const doubled = computed(() => count.value * 2)  // lazy, zero overhead when not read
     ```
   - computed is free when not subscribed — effects always run

2. **Reactive leak prevention** — Explain that computed() and effect() created outside an owner scope are never automatically disposed:
   - Dev mode warns: "[Streem] computed() created without an active owner scope"
   - Two safe patterns: (a) create inside a component function (automatic owner from render()), (b) wrap in createRoot() and call dispose() when done
   - Show the problematic pattern and the fix:
     ```typescript
     // PROBLEMATIC — created at module level, never disposed:
     const doubled = computed(() => count.value * 2)  // dev warning fires

     // SAFE PATTERN A — inside a component (automatic owner):
     function MyComponent() {
       const doubled = computed(() => count.value * 2)
       return <p>{doubled}</p>
     }

     // SAFE PATTERN B — explicit root for module-level reactive work:
     const root = createRoot(() => {
       const doubled = computed(() => count.value * 2)
       effect(() => { document.title = String(doubled()) })
     })
     // Later when done:
     root.dispose()
     ```

3. **Cleanup patterns** — Show the three cleanup hooks and when to use each:
   - `onCleanup()` inside effect() — runs before next effect re-run AND on dispose. Use for: clearing timers, removing event listeners created inside the effect
   - `onMount()` returning a function — cleanup runs on component unmount. Use for: one-time setup (timers, subscriptions) that should stop when the component leaves the DOM
   - `createRoot().dispose()` — explicit teardown for long-lived reactive trees outside components
   - Full example with onMount + cleanup:
     ```typescript
     function LiveClock() {
       const time = signal(new Date().toLocaleTimeString())

       onMount(() => {
         const id = setInterval(() => {
           time.set(new Date().toLocaleTimeString())
         }, 1000)
         return () => clearInterval(id)  // fires on unmount
       })

       return <span>{time}</span>
     }
     ```
   - Full example with effect + onCleanup:
     ```typescript
     effect(() => {
       const handler = (e: KeyboardEvent) => console.log(e.key)
       window.addEventListener('keydown', handler)
       onCleanup(() => window.removeEventListener('keydown', handler))
     })
     ```

4. **Signal granularity** — Explain fine-grained vs coarse-grained signals and when each is correct:
   - Fine-grained (separate signals per field): each subscriber only re-runs when its specific field changes. Best for independent values that update at different rates (e.g. price vs volume in a trading feed).
   - Coarse-grained (one signal holding an object): simpler for values that always update together. Any field change notifies all subscribers.
   - Show both:
     ```typescript
     // Fine-grained — subscribers only re-run when their signal changes:
     const price = signal(0)
     const volume = signal(0)
     const priceLabel = computed(() => `$${price.value.toFixed(2)}`)
     const volumeLabel = computed(() => `${volume.value.toLocaleString()} shares`)

     // Coarse-grained — one update notifies all subscribers:
     const ticker = signal({ price: 0, volume: 0 })
     // Updating one field requires spreading the whole object:
     ticker.set({ ...ticker.value, price: 42.5 })
     ```
   - Rule of thumb: prefer fine-grained for values that update independently; prefer coarse-grained for values that always change together (forms, config objects).
