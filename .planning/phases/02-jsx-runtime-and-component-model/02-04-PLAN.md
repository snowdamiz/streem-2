Purpose: ErrorBoundary provides fault isolation for synchronous render errors. This is the correct Phase 2 boundary — the synchronous error path (ErrorBoundary catches render errors) and the pending state path (Suspense catches thrown Promises) are both fully implemented here.

  <behavior>
    **ErrorBoundary:**
    Props: { fallback: (err: unknown, reset: () => void) => Node; children: Node | (() => Node) }

    Cases:
      children render without error → normal DOM output, fallback never shown
      child throws Error during render → fallback(err, reset) called, fallback DOM shown
      reset() called → re-render children from scratch (dispose stale scope, retry)
      child throws Promise → ErrorBoundary RE-THROWS the Promise (does not call fallback)
      fallback can itself throw → that propagates up (no recursive catch)

    **Suspense:**
    Props: { fallback: Node | (() => Node); children: Node | (() => Node) }

    Cases:
      children render without throwing → children DOM shown, fallback never shown
      child throws Promise → fallback shown immediately; Promise.then() schedules retry
      Promise resolves → children re-rendered; fallback replaced with children DOM
      Promise rejects → rejection error is logged via console.error (Phase 2 scope);
        full async propagation to ErrorBoundary is deferred to Phase 3
      child throws a non-Promise Error → Suspense re-throws it (propagates to ErrorBoundary above)
      multiple children throw Promises → show fallback until ALL resolve (simple: track a counter)

    **Component interaction (MUST be correct):**
    When ErrorBoundary wraps Suspense wraps children:
    1. Child throws Promise → Suspense catches it (instanceof Promise check in Suspense's try/catch)
    2. Suspense shows fallback
    3. Promise resolves → Suspense retries, children render normally
    4. If child throws Error (not Promise) → Suspense re-throws → ErrorBoundary catches
    5. If Promise rejects → console.error in Phase 2 (Phase 3 wires full ErrorBoundary propagation)
  </behavior>
  <implementation>
    **ErrorBoundary() implementation:**
    ```typescript
    export function ErrorBoundary(props: {
      fallback: (err: unknown, reset: () => void) => Node | Node[] | null
      children: Node | Node[] | (() => Node | Node[] | null)
    }): Node {
      const anchor = document.createComment('ErrorBoundary')
      let currentDispose: (() => void) | null = null
      let currentNodes: Node[] = []

      const removeCurrentNodes = () => {
        currentNodes.forEach(n => n.parentNode?.removeChild(n))
        currentNodes = []
        currentDispose?.()
        currentDispose = null
      }

      const insertNodes = (nodes: Node | Node[] | null | undefined) => {
        if (!nodes) return
        const arr = Array.isArray(nodes) ? nodes : [nodes]
        currentNodes = arr
        arr.forEach(n => anchor.parentNode?.insertBefore(n, anchor))
      }

      const renderChildren = (): Node | Node[] | null => {
        try {
          let childResult: Node | Node[] | null | undefined = null
          createRoot((d) => {
            currentDispose = d
            const child = typeof props.children === 'function' ? props.children() : props.children
            childResult = child ?? null
          })
          return childResult
        } catch (err) {
          if (err instanceof Promise) throw err  // CRITICAL: re-throw Promises to Suspense
          currentDispose?.()
          currentDispose = null

          const reset = () => {
            removeCurrentNodes()
            const freshNodes = renderChildren()
            insertNodes(freshNodes)
          }
          return props.fallback(err, reset)
        }
      }

      // Initial render — not yet in DOM, just return anchor
      // Actual insertion happens when parent inserts the returned node
      // For simplicity in Phase 2: render immediately and store
      let result: Node | Node[] | null = null
      try {
        result = renderChildren()
      } catch (err) {
        if (err instanceof Promise) throw err
        result = props.fallback(err, () => {})
      }
      // Return anchor; component handles its own DOM management via anchor
      // Note: For Phase 2, ErrorBoundary inlines its DOM before returning anchor
      // The caller (h() via createRoot) sees the anchor node
      onCleanup(() => { currentDispose?.() })
      return anchor
    }
    ```

    DESIGN NOTE: The above ErrorBoundary returns an anchor comment. But the challenge is
    that the PARENT hasn't yet inserted the anchor into the DOM when renderChildren() is called.
    For Phase 2, use a simpler approach: ErrorBoundary renders synchronously and returns
    the result nodes directly (not via an anchor). Use an anchor only as a stable reference.

    Recommend this simpler design for Phase 2:
    - ErrorBoundary renders children immediately in try/catch
    - Returns the resulting Node (child or fallback)
    - For reset(): the caller must re-render — but without an anchor in the DOM, we can't swap
    - Solution: ErrorBoundary returns a DocumentFragment or wraps in a span container? No.
    - Best approach: return an anchor comment, and initialize DOM lazily in a queueMicrotask
      OR accept that reset() requires the parent to re-run.

    Actually the simplest correct Phase 2 approach: ErrorBoundary uses an anchor comment
    as the stable return node, and BOTH the initial render and reset insert before that anchor.
    The parent receives and inserts the anchor into the DOM. After the anchor is in the DOM,
    ErrorBoundary's initial DOM is also inserted (siblings before anchor). This works because
    h() / appendChildren() inserts the returned node after createRoot completes — and within
    that createRoot, ErrorBoundary has already run synchronously. The timing:
    1. h(ErrorBoundary, props) → createRoot → ErrorBoundary() runs
    2. Inside ErrorBoundary(): renderChildren() runs, DOM is... not yet inserted
    3. ErrorBoundary returns anchor comment
    4. h() returns anchor to parent
    5. Parent appends anchor to DOM
    6. But ErrorBoundary's child DOM is NOT yet in DOM at this point!

    Resolution: Use the same deferred pattern as Show/For — ErrorBoundary does NOT pre-insert
    its children. Instead, it returns the anchor, and manages insertion/removal of sibling nodes
    relative to that anchor. The initial render must happen AFTER the anchor is in the DOM.
    Use `queueMicrotask` for initial render in ErrorBoundary.

    OR: simpler — just insert children synchronously into a DocumentFragment, and h() appends
    the fragment. But then reset() can't work without anchor.

    **Recommended Phase 2 approach:** Keep it simple.
    - ErrorBoundary wraps children render in try/catch
    - Returns the child Node directly (or fallback Node on error)
    - reset() is provided to the fallback function but triggers parent-scoped re-render
      rather than in-place DOM swap (in-place swap requires anchor infrastructure, Phase 3+)
    - This is sufficient for COMP-06 which says "catch errors and render fallback UI"
    - In-place swap with reset is a Phase 2+ enhancement (Phase 2 defines the protocol)

    This avoids the anchor/timing complexity for Phase 2. Suspense requires anchor-based
    DOM management because it MUST swap async (can't do synchronously).

    **Suspense() implementation:**
    Suspense MUST use anchor-based management because it swaps asynchronously.
    ```typescript
    export function Suspense(props: {
      fallback: Node | Node[] | (() => Node | Node[] | null)
      children: Node | Node[] | (() => Node | Node[] | null)
    }): Node {
      const anchor = document.createComment('Suspense')
      let pendingCount = 0
      let currentNodes: Node[] = []

      const insertBefore = (nodes: Node | Node[] | null | undefined) => {
        if (!nodes) return
        const arr = Array.isArray(nodes) ? nodes : [nodes]
        currentNodes.forEach(n => n.parentNode?.removeChild(n))
        currentNodes = arr
        // defer until anchor is in DOM
        queueMicrotask(() => {
          arr.forEach(n => anchor.parentNode?.insertBefore(n, anchor))
        })
      }

      const getFallback = () => {
        const fb = typeof props.fallback === 'function' ? props.fallback() : props.fallback
        return fb
      }

      const tryRenderChildren = (): { success: boolean } => {
        try {
          const child = typeof props.children === 'function' ? props.children() : props.children
          insertBefore(child)
          return { success: true }
        } catch (err) {
          if (!(err instanceof Promise)) {
            throw err  // Non-Promise errors propagate to ErrorBoundary
          }
          // Pending: show fallback
          pendingCount++
          insertBefore(getFallback())
          err.then(
            () => {
              pendingCount--
              if (pendingCount === 0) tryRenderChildren()
            },
            (rejectionError) => {
              // Phase 2: log rejected Promise. Full async propagation to ErrorBoundary
              // is implemented in Phase 3 alongside createResource.
              console.error('Streem <Suspense>: resource rejected:', rejectionError)
            }
          )
          return { success: false }
        }
      }

      queueMicrotask(() => {
        tryRenderChildren()
      })

      onCleanup(() => {
        currentNodes.forEach(n => n.parentNode?.removeChild(n))
      })

      return anchor
    }
    ```

    NOTE: The above Suspense uses queueMicrotask for initial render (after anchor is in DOM).
    Tests must account for this — use `await new Promise(resolve => queueMicrotask(resolve))`
    or `await new Promise(resolve => setTimeout(resolve, 0))` to flush microtasks before asserting.

    **Test file strategy:**
    Write tests using happy-dom. For Suspense async tests, use `await new Promise(r => queueMicrotask(r))`.
    For ErrorBoundary tests, synchronous tests are sufficient.
